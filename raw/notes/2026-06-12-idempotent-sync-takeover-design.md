# 이종 시스템 전환기 중복 발송 방지 — 장부 기반 멱등 동기화 엔진 설계

> 병원 CRM 리뉴얼 프로젝트에서, 레거시 시스템과 신규 시스템이 **같은 문자 발송 큐를 공유하면서 발송 이력 장부는 분리**된 상황의 전환(cutover) 문제를 해결한 설계 기록.
> 키워드: 멱등성(idempotency), 상태 수렴(reconciliation), 분산 락, 무중단 전환, 레거시 마이그레이션

---

## 1. 배경

병원용 CRM(VB6 레거시)을 Electron + TypeScript 로 리뉴얼하면서, 예약 안내·검진 리마인드 등을 보내는 **자동발송 엔진**을 새로 만들었다. 구조상 제약은 다음과 같았다.

- 레거시와 신규는 **같은 EMR(원무 DB)** 을 읽고, **같은 문자 벤더 큐 테이블**(SQL 테이블 — 벤더가 폴링해서 발송)에 INSERT 한다.
- 그러나 "무엇을 보냈는지" 기록하는 **장부는 서로 분리**되어 있다 (레거시: EMR 내 테이블 / 신규: 자체 스키마). 서로의 발송 이력을 모른다.
- 발송 모델은 즉시 발송이 아니라 **선예약(pre-enqueue)**: 예약 등록 시점에 "D-1 오전 10시 발송" 행을 벤더 큐에 미리 넣고, 이후 예약이 변경·취소되면 그 행을 갱신·삭제하는 방식.

이 상태에서 병원이 레거시 → 신규로 전환하면 세 가지 사고가 가능하다.

| 시나리오 | 결과 |
|---|---|
| 레거시가 미래 선예약을 박아둔 뒤 전환, 신규도 같은 예약을 등록 | **환자에게 같은 문자 2회 발송** |
| 레거시 선예약이 남은 채 환자가 예약 취소 — 레거시는 꺼져서 취소 동기화 불가 | **취소한 환자에게 발송** (고아 행) |
| 전환 당일 오전 레거시가 이미 발송, 오후에 신규가 켜져 재등록 | **2회 발송** |

문자 오발송은 환자 컴플레인과 병원 신뢰 문제로 직결되므로, "운영 절차를 잘 지키면 된다"가 아니라 **코드 수준에서 구조적으로 차단**해야 했다.

## 2. 기반 설계 — 장부 기반 멱등 동기화 엔진

전환 문제를 풀기 전에, 신규 엔진 자체가 멱등하게 설계되어 있다. 1분 주기 틱(tick)으로 도는 엔진이 "몇 번을 다시 실행해도 결과가 같다"를 보장하는 장치는 3겹이다.

### 2-1. 멱등 키 — 한 엔티티 = 장부 한 행

모든 발송 대상을 자연 키로 정규화한다.

```
예약/예약당일: 가맹코드|rule|예약ID          ← 예약 하나당 평생 1행
수술:         가맹코드|surgery|일련번호
처방코드:     가맹코드|code|접수/처방/코드ID
생일:         가맹코드|birthday|챠트번호|YYYY     ← 연 1회 보장이 키에 내장
신환:         가맹코드|newpatient|챠트번호|YYYYMMDD ← 일 1회 보장이 키에 내장
```

같은 예약이 천 번 감지돼도 키가 같으므로 장부 행은 하나다. 장부 쓰기는 upsert(있으면 UPDATE, 없으면 INSERT)라 행이 증식하지 않는다. "연 1회·일 1회" 같은 빈도 정책을 별도 카운터가 아니라 **키 설계에 내장**한 것이 포인트 — 정책 위반이 코드 버그가 아니라 키 충돌로 막힌다.

### 2-2. 순수 판정 함수 — 차이가 없으면 no-op

매 틱마다 EMR 에서 현재 상태(후보)를 다시 뽑고, 순수 함수 `decideSync(현재 상태, 장부 행)` 가 할 일을 결정한다.

| 현재 상태 vs 장부 | 결정 |
|---|---|
| 장부에 없음 | `insert` — 벤더 큐 등록 |
| 내용·시각·타입·번호 전부 동일 | `skip` — 아무것도 안 함 |
| 메시지/시각/번호 변경 | `update` — 벤더 행 갱신 |
| SMS↔LMS 타입 변경 | `replace` — 삭제 성공 시에만 재등록 |
| EMR 취소 + 벤더 대기 중 | `cancel` — 벤더 행 삭제 |
| 이미 발송 완료(closed) | `skip` — 발송 후엔 어떤 변경에도 재발송 금지 |

평상시 틱의 실제 모습은 "전부 skip"이다. 변화가 생기면 그 차이만큼만 벤더에 반영하고 다시 전-skip 상태로 수렴한다. 즉 **"보냈는지 기억하기"가 아니라 "원하는 상태와 실제 상태의 차이를 0으로 만들기"** — 쿠버네티스 reconciliation loop 와 같은 개념을 문자 발송에 적용한 것이다.

판정 함수를 순수 함수로 분리한 덕에 분기 전수(취소 재활성, 발송 후 타입 변경, failed 재시도 등)를 DB 없이 단위 테스트로 고정했다. 참고로 레거시는 매 틱 무조건 재제출(UPDATE)했는데, 신규는 무변경 왕복을 생략해 같은 결과를 더 적은 부하로 낸다. 또 레거시에는 "발송 완료 후 타입이 바뀌면 새 INSERT 가 나가는" 이중발송 결함이 있었는데, closed 상태를 명시해 의도적으로 회피했다.

### 2-3. 실행 안전망

- **건별 격리**: 한 건이 실패해도 장부가 안 바뀌므로 다음 틱에 같은 판정이 다시 나와 자동 재시도. 전체 틱은 계속 진행.
- **다중 설치 락**: 병원에 앱이 여러 대 설치돼도 `sp_getapplock`(SQL Server 어플리케이션 락)으로 틱당 1대만 실행. 동시 insert 경합을 원천 차단하고, 실행 중이던 PC가 꺼지면 다른 PC가 자동 인계(페일오버).
- **replace 순서 보장**: 기존 벤더 행 삭제가 성공해야만 재등록. 삭제 0건 = 이미 발송됨 = closed 처리하고 새로 넣지 않는다.

## 3. 전환 설계 — 틱 내 "레거시 인수(takeover)" 단계

위 멱등 엔진이 있었기에, 전환 문제는 별도 마이그레이션 도구가 아니라 **틱에 단계 하나를 추가**하는 것으로 풀렸다.

### 3-1. 핵심 아이디어: 삭제가 아니라 "엔티티 단위 교체"

레거시 장부에는 각 선예약의 벤더 큐 행 번호(SEQ)가 적혀 있다. 이를 읽으면(읽기 전용 — 레거시 장부에 쓰지 않는다) 레거시가 넣은 벤더 행을 **추정이 아니라 정확 지목**으로 다룰 수 있다. 매 틱:

1. 레거시 장부에서 "미발송 + 벤더 행 있음 + **우리가 켠 룰에 해당**" 잔존 행 조회
2. 각 행의 엔티티(예약ID 등)를 EMR 원본에서 재판정:
   - **예약이 살아있음** → 우리 메시지를 벤더 큐에 **먼저 upsert → 성공 시에만** 레거시 행 삭제. 순서 고정으로 발송 공백 제로.
   - **예약이 취소됨** → 레거시 행만 삭제. 이것이 고아 행 제거 — 취소 환자 오발송 차단.
   - **판정 불가 / 대체 불가**(미구현 타입, 꺼진 룰) → 건드리지 않음 + 로그. 보수적 기본값.
3. 우리 enqueue 직전 공통 가드: 레거시 장부에 같은 (룰, 엔티티)가 **발송 완료**로 있으면 skip — 전환 당일 "오전에 레거시가 이미 보낸 건"의 재발송 차단.

"지웠는데 다시 안 보내는" 사고는 구조적으로 불가능하다 — **다시 까는 게 먼저고 지우는 게 나중이며, 못 까는 건 안 지우기 때문**. 실패 방향도 안전하다: upsert 실패 시 레거시 행이 남아 발송 자체는 보장(누락 없음), 삭제 실패 시 최악이 중복 1회이며 로그로 표면화 후 다음 틱 재시도.

### 3-2. 의도적 트레이드오프 — 1회성 스크립트 대신 매 틱 실행

직관적으로는 "전환 시 한 번만 정리"가 맞아 보이지만 매 틱을 택했다.

- **비용이 사실상 0**: 잔존 0건이면 SELECT 1회로 끝. 엔진은 어차피 매분 룰 감지 쿼리 6개를 실행 중.
- **자가수렴**: 부분 실패는 다음 틱이 재시도. "인수 완료" 플래그 테이블과 그 리셋 시나리오("레거시가 실수로 재가동되면 플래그를 누가 푸나") 관리 코드가 통째로 불필요.
- **절차 실수 내성**: 운영 절차(레거시 룰 끄기 → 신규 켜기)가 어긋나거나 레거시가 재가동돼도 자동 수습. 병원 수가 늘수록 절차 누락 확률은 누적되므로, 방어를 절차가 아닌 코드에 둔다.

같은 이유로 "앱 시작 시 1회 정리"도 기각 — 지우는 주체(앱)와 다시 까는 주체(발송 엔진)가 분리되면, 룰이 아직 꺼진 상태에서 지워 버려 메시지가 증발하는 창이 생긴다. **지우기와 다시 깔기를 같은 엔진의 같은 사이클에 묶는 것**이 공백 제로의 본질이다.

### 3-3. 관측성

모든 인수 동작을 사유와 함께 로그로 남긴다: `takeover replace|remove|skip rule=<룰> entity=<ID> seq=<SEQ> reason=<사유>`, 틱마다 잔존 카운트(`takeover remaining=N`)로 수렴 추이를 확인. 교체된 건은 신규 장부에 정상 행으로 남아 이후 취소·변경 동기화를 자동 승계한다.

## 4. 요약 — 설계 원칙

| 원칙 | 적용 |
|---|---|
| 멱등 키에 정책을 내장 | 연 1회/일 1회를 카운터가 아니라 키 설계로 보장 |
| 기억이 아니라 수렴 | 매 틱 "원하는 상태 - 실제 상태 = 0" reconciliation. 평상시 = 전부 no-op |
| 순수 함수로 판정 분리 | 분기 전수를 DB 없는 단위 테스트로 고정 |
| 실패는 안전한 방향으로 | 재적재 먼저·삭제 나중, 못 까는 건 안 지움 — 누락 불가, 최악이 중복 1회 + 로그 |
| 방어는 절차가 아니라 코드에 | 전환 절차 실수·레거시 재가동도 틱이 자동 수습 |
| 레거시는 관찰자 | 레거시 장부는 읽기 전용 — 기존 시스템 동작을 변경하지 않고 전환 |
