# 나만의 세컨 브레인: 옵시디언 기반 LLM 위키 구축기

**작가:** 곰씨네 IT 블로그
**발행일:** 2일 전
**읽는 시간:** 10분

---

## 요점 정리

- 여러 AI 도구 사용 시 컨텍스트를 반복 설명하는 번거로움 해결 필요
- 옵시디언, 깃허브, 클로드 코드로 개인 LLM 위키 구축
- AI가 작성·관리하는 위키로 맞춤형 답변 품질 향상

---

## 본문

### LLM 위키를 만든 이유

**여러 곳에 흩어진 컨텍스트**

사용자들은 클로드, 챗지피티, 제미나이 등 여러 AI 도구를 병행합니다. 저자는 "기획은 클로드, 검색은 챗지피티, 코드는 클로드 코드" 식으로 도구를 분산하여 사용하다 보니 각 플랫폼마다 상황을 반복 설명해야 했습니다. 구글 킵, 노션, 구글 워크스페이스, 깃허브 등에 정보가 흩어져 있어 "AI가 온전한 맥락을 파악하기 어려웠다"고 설명합니다.

**분산된 정보의 통합**

저자는 안드레 카파시의 'LLM 위키' 개념을 적용하여 모든 개인 데이터를 한 곳에 모으기로 결정했습니다. 도구 선택 기준은 빠른 속도, 안정적 동기화, 워크플로우 적합성이었으며, 최종적으로 옵시디언을 선택했습니다.

### 환경 세팅

**옵시디언 구성**

미니멀 테마를 적용하고 Git 플러그인(동기화용), 터미널 플러그인(클로드 코드 실행용) 등을 설치했습니다. 웹 클리퍼로 데스크톱 자료를, 모바일 공유 기능으로 스마트폰 자료를 옵시디언 볼트로 일원화했습니다.

**깃허브 동기화**

Private Repository를 클라우드 백업으로 활용하며, 모바일은 Working Copy 앱(약 $20 일회성 결제)을 통해 동기화합니다.

**클로드 코드 활용**

CLAUDE.md를 루트 디렉터리에 두고 "새 세션 시마다 시스템 컨텍스트로 자동 주입"되도록 설정했습니다. SessionStart Hook으로 운영 규칙과 위키 인덱스를 강제로 참조하게 하여 AI의 일관성을 확보합니다.

### 위키 설계 및 활용

**카테고리 구성**

정체성(Identity), 생각(Thoughts), 목표(Goals), 역사(History), 사람(People), 자산(Assets), 일(Works)로 분류합니다.

**자료 인입(Ingest)**

클로드 코드의 /ingest 스킬이 새 자료를 읽고 적절한 위키 페이지로 합성·연결합니다. 정규식으로 민감 정보(비밀번호, 카드번호 등)를 감지하고 마스킹 처리합니다.

**점검 및 정리(Lint)**

/lint 스킬은 깨진 링크, 고아 페이지, 모순된 정보를 자동 감지합니다. 클로드 스케쥴 기능으로 매일 자동 실행됩니다.

**활용(Query)**

"/query 스킬로 전체 위키를 컨텍스트로 깔고 질문"하면 "나의 철학, 상황, 프로젝트 상황까지 모두 고려한 맞춤형 답변"을 받을 수 있습니다.

### 실제 경험과 한계

**긍정적 결과**

법인 설립, 프로젝트 우선순위 결정 같은 복잡한 의사결정에서 "일반론적 답변"이 아닌 "개인 상황 최적화 답변"을 받을 수 있었습니다.

**한계점**

- 이미지 텍스트 인식 부정확 (직접 수정 필요)
- 민감 정보 증가로 보안 우려 증가
- 깃허브 해킹 시 정보 전체 노출 위험
- 협업 시 개인정보 문제로 팀 LLM 위키 별도 필요 (이중 작업)

## 결론

저자는 LLM 위키 구축을 "컨텍스트를 자산처럼 쌓아가는 장기 과정"으로 정의합니다. 다만 "처음부터 완벽한 구조를 만들려는 욕심"은 내려놓고 "점진적으로 개선"해야 한다고 강조합니다.
