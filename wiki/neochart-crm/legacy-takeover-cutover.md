# 레거시 인수(takeover) — 전환기 중복 발송 방지

레거시(VB6)와 신규 CRM이 **같은 문자 벤더 큐를 공유하면서 발송 이력 장부는 분리**된 상황에서, 병원의 레거시 → 신규 전환(cutover) 시 중복·오발송을 코드 수준에서 구조적으로 차단한 설계.

## 문제 — 전환 시 가능한 3가지 사고

| 시나리오 | 결과 |
|---|---|
| 레거시가 미래 선예약을 박아둔 뒤 전환, 신규도 같은 예약 등록 | 같은 문자 2회 발송 |
| 레거시 선예약 잔존 + 환자 예약 취소 (레거시는 꺼져서 취소 동기화 불가) | 취소한 환자에게 발송 (고아 행) |
| 전환 당일 오전 레거시 발송 완료, 오후 신규 가동 후 재등록 | 2회 발송 |

문자 오발송은 환자 컴플레인·병원 신뢰 문제로 직결 → "운영 절차를 잘 지키면 된다"가 아니라 코드로 차단.

## 해법 — 틱 내 takeover 단계 (엔티티 단위 교체)

[[autosend-sync-engine]]이 멱등하게 설계되어 있었기에, 별도 마이그레이션 도구 없이 **틱에 단계 하나를 추가**하는 것으로 해결. 레거시 장부에 적힌 벤더 큐 행 번호(SEQ)를 **읽기 전용**으로 참조해, 레거시가 넣은 행을 추정이 아닌 정확 지목으로 다룬다. 매 틱:

1. 레거시 장부에서 "미발송 + 벤더 행 있음 + 우리가 켠 룰 해당" 잔존 행 조회
2. 각 행의 엔티티를 EMR 원본에서 재판정:
   - **예약 살아있음** → 우리 메시지를 벤더 큐에 **먼저 upsert, 성공 시에만** 레거시 행 삭제 (발송 공백 제로)
   - **예약 취소됨** → 레거시 행만 삭제 (고아 행 제거 — 취소 환자 오발송 차단)
   - **판정 불가/대체 불가** (미구현 타입, 꺼진 룰) → 건드리지 않고 로그 (보수적 기본값)
3. enqueue 직전 공통 가드: 레거시 장부에 같은 (룰, 엔티티)가 **발송 완료**로 있으면 skip (전환 당일 오전 발송분 재발송 차단)

**실패 방향이 안전하다**: "다시 까는 게 먼저, 지우는 게 나중, 못 까는 건 안 지움" — upsert 실패 시 레거시 행이 남아 발송은 보장(누락 불가), 삭제 실패 시 최악이 중복 1회 + 로그 표면화 + 다음 틱 재시도.

## 의도적 트레이드오프 — 1회성 스크립트 대신 매 틱 실행

- **비용 ≈ 0**: 잔존 0건이면 SELECT 1회. 엔진은 어차피 매분 룰 감지 쿼리 6개 실행 중.
- **자가수렴**: 부분 실패는 다음 틱이 재시도. "인수 완료" 플래그 테이블과 리셋 시나리오 관리가 통째로 불필요.
- **절차 실수 내성**: 운영 절차(레거시 룰 끄기 → 신규 켜기)가 어긋나거나 레거시가 재가동돼도 자동 수습. 병원 수가 늘수록 절차 누락 확률이 누적되므로 방어를 절차가 아닌 코드에 둠.
- "앱 시작 시 1회 정리"도 기각: 지우는 주체(앱)와 다시 까는 주체(발송 엔진)가 분리되면 메시지 증발 창이 생김. **지우기와 다시 깔기를 같은 엔진의 같은 사이클에 묶는 것**이 공백 제로의 본질.

## 관측성

- 모든 인수 동작을 사유와 함께 로그: `takeover replace|remove|skip rule=<룰> entity=<ID> seq=<SEQ> reason=<사유>`
- 틱마다 잔존 카운트 `takeover remaining=N`으로 수렴 추이 확인.
- 교체된 건은 신규 장부에 정상 행으로 남아 이후 취소·변경 동기화를 자동 승계.

## 설계 원칙 (원문 요약)

| 원칙 | 적용 |
|---|---|
| 멱등 키에 정책 내장 | 연 1회/일 1회를 카운터가 아닌 키 설계로 |
| 기억이 아니라 수렴 | 매 틱 reconciliation, 평상시 = 전부 no-op |
| 순수 함수로 판정 분리 | 분기 전수를 DB 없는 단위 테스트로 고정 |
| 실패는 안전한 방향으로 | 누락 불가, 최악이 중복 1회 + 로그 |
| 방어는 절차가 아니라 코드에 | 절차 실수·레거시 재가동도 틱이 자동 수습 |
| 레거시는 관찰자 | 레거시 장부는 읽기 전용 — 기존 시스템 무변경 전환 |
