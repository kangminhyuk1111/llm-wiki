# autosend 멱등 동기화 엔진

신규 자동발송 엔진(예약 안내·검진 리마인드 SMS)의 핵심 설계. 병원 CRM을 VB6 레거시에서 Electron + TypeScript로 리뉴얼하며 새로 만든 엔진으로, 1분 주기 틱(tick)이 "몇 번을 다시 실행해도 결과가 같다"(멱등성)를 3겹 장치로 보장한다.

발송 모델은 즉시 발송이 아니라 **선예약(pre-enqueue)**: 예약 등록 시점에 "D-1 오전 10시 발송" 행을 문자 벤더 큐 테이블(SQL, 벤더가 폴링)에 미리 넣고, 예약 변경·취소 시 그 행을 갱신·삭제한다.

## 1겹 — 멱등 키: 한 엔티티 = 장부 한 행

모든 발송 대상을 자연 키로 정규화한다. 같은 예약이 천 번 감지돼도 키가 같으므로 장부 행은 하나 (upsert).

```
예약/예약당일: 가맹코드|rule|예약ID            ← 예약 하나당 평생 1행
수술:         가맹코드|surgery|일련번호
처방코드:     가맹코드|code|접수/처방/코드ID
생일:         가맹코드|birthday|챠트번호|YYYY      ← 연 1회 보장이 키에 내장
신환:         가맹코드|newpatient|챠트번호|YYYYMMDD ← 일 1회 보장이 키에 내장
```

핵심 발상: "연 1회·일 1회" 빈도 정책을 카운터가 아니라 **키 설계에 내장** — 정책 위반이 코드 버그가 아니라 키 충돌로 막힌다.

## 2겹 — 순수 판정 함수: 차이가 없으면 no-op

매 틱 EMR에서 현재 상태를 다시 뽑고, 순수 함수 `decideSync(현재 상태, 장부 행)`가 결정한다:

| 현재 상태 vs 장부 | 결정 |
|---|---|
| 장부에 없음 | `insert` — 벤더 큐 등록 |
| 전부 동일 | `skip` |
| 메시지/시각/번호 변경 | `update` |
| SMS↔LMS 타입 변경 | `replace` — 삭제 성공 시에만 재등록 |
| EMR 취소 + 벤더 대기 중 | `cancel` — 벤더 행 삭제 |
| 발송 완료(closed) | `skip` — 발송 후엔 어떤 변경에도 재발송 금지 |

평상시 틱은 "전부 skip"이고, 변화가 생기면 그 차이만큼만 반영해 다시 전-skip으로 수렴한다. **"보냈는지 기억하기"가 아니라 "원하는 상태와 실제 상태의 차이를 0으로 만들기"** — Kubernetes reconciliation loop 개념의 문자 발송 적용.

- 순수 함수 분리 덕에 분기 전수(취소 재활성, 발송 후 타입 변경, failed 재시도)를 DB 없이 단위 테스트로 고정.
- 레거시 대비: 레거시는 매 틱 무조건 재제출(UPDATE)했고 "발송 완료 후 타입 변경 시 새 INSERT가 나가는" 이중발송 결함이 있었음 — 신규는 무변경 왕복 생략 + closed 상태 명시로 회피.

## 3겹 — 실행 안전망

- **건별 격리**: 한 건 실패 시 장부가 안 바뀌므로 다음 틱에 같은 판정이 재현되어 자동 재시도. 전체 틱은 계속 진행.
- **다중 설치 락**: `sp_getapplock`(SQL Server 어플리케이션 락)으로 틱당 1대만 실행. 동시 insert 경합 차단 + 실행 PC가 꺼지면 다른 PC가 자동 인계(페일오버). ([[autosend-heartbeat]]·[[db-auto-migration]]의 "락 승자"가 이 장치)
- **replace 순서 보장**: 기존 벤더 행 삭제가 성공해야만 재등록. 삭제 0건 = 이미 발송됨 = closed 처리하고 새로 넣지 않음.
