# 옵시디언 기반 LLM 위키 구축 사례 (곰씨네 IT 블로그)

Karpathy의 LLM 위키 개념을 Obsidian + GitHub + Claude Code로 구현한 개인 사례. 요즘IT 기고글(2026-06 추정) 기반. 이 저장소와 같은 패턴의 외부 사례라 구성 비교와 개선 아이디어 출처로 유용하다.

## 구축 동기

- 클로드·챗GPT·제미나이 등 여러 AI 도구를 용도별로 병행("기획은 클로드, 검색은 챗지피티, 코드는 클로드 코드")하다 보니 플랫폼마다 상황을 반복 설명해야 했음.
- 구글 킵, 노션, 구글 워크스페이스, 깃허브에 정보가 흩어져 "AI가 온전한 맥락을 파악하기 어려웠다" → 모든 개인 데이터를 한 곳에 통합.
- 도구 선택 기준: 빠른 속도, 안정적 동기화, 워크플로우 적합성 → Obsidian 선택.

## 환경 구성

| 구성 요소 | 선택 | 비고 |
|---|---|---|
| 에디터/뷰어 | Obsidian (미니멀 테마) | Git 플러그인(동기화), 터미널 플러그인(Claude Code 실행) |
| 수집 | Obsidian Web Clipper(데스크톱) + 모바일 공유 | 볼트로 일원화 |
| 백업/동기화 | GitHub Private Repository | 모바일은 Working Copy 앱(~$20 일회성) |
| 에이전트 | Claude Code | 루트 CLAUDE.md가 매 세션 자동 주입, SessionStart Hook으로 운영 규칙·위키 인덱스 강제 참조 |

## 위키 설계

- 카테고리: 정체성(Identity) / 생각(Thoughts) / 목표(Goals) / 역사(History) / 사람(People) / 자산(Assets) / 일(Works) — 삶 전반을 다루는 분류.
- 워크플로 스킬 3종: `/ingest`(자료 합성·연결 + **정규식으로 민감 정보(비밀번호·카드번호) 감지·마스킹**), `/lint`(깨진 링크·고아 페이지·모순 감지, **Claude 스케줄 기능으로 매일 자동 실행**), `/query`(전체 위키를 컨텍스트로 깔고 질문).

## 성과와 한계

- 성과: 법인 설립, 프로젝트 우선순위 같은 복잡한 의사결정에서 일반론이 아닌 "개인 상황 최적화 답변"을 얻음.
- 한계: ① 이미지 텍스트 인식 부정확(직접 수정 필요) ② 민감 정보 축적에 따른 보안 우려 ③ GitHub 해킹 시 전체 노출 위험 ④ 협업 시 개인정보 문제로 팀 위키를 따로 운영해야 하는 이중 작업.
- 저자의 결론: LLM 위키는 "컨텍스트를 자산처럼 쌓아가는 장기 과정" — 처음부터 완벽한 구조를 욕심내지 말고 점진적으로 개선하라.

## 이 저장소와의 비교 (추정/관찰)

- 공통: Obsidian 뷰어, GitHub 동기화, CLAUDE.md 스키마, ingest/lint/query 3종 스킬 구성이 사실상 동일.
- 차이: 이 사례는 ① 민감 정보 자동 마스킹 ② lint 매일 자동 실행(스케줄) ③ 삶 전반의 카테고리 체계를 갖춤 — 셋 다 이 저장소에는 아직 없는 요소로, 도입 검토 후보다 (특히 ①은 이 저장소가 public이면 중요).
